```javascript
async prompt(text, opts) {
  // 1. 检查 session 存在
  if (!this._session) throw new Error("没有活跃的 session");
  // this._session 在 server/index.js 启动时已通过 engine.createSession() 创建

  // 2. 标记 session 已开始
  this._sessionStarted = true;

  // 3. 构建 prompt 选项（无图片，promptOpts = undefined）
  const promptOpts = opts?.images?.length ? { images: opts.images } : undefined;
  // → undefined

  // 4. 调用 Pi SDK 的 session.prompt()
  await this._session.prompt(text, promptOpts);
  // 这一步会：
  //   a. 将 text 追加到 session messages 数组（role: "user"）
  //   b. 写入 JSONL 文件
  //   c. 调用 AI 模型 API
  //   d. 流式接收响应，逐 token 触发 session 事件

  // 5. 通知记忆系统
  const sp = this._session.sessionManager?.getSessionFile?.();
  // → "/Users/laoye/.hanako/agents/hanako/sessions/2026-03-18T15-30-00.jsonl"
  if (sp) this._d.getAgent()._memoryTicker?.notifyTurn(sp);
}
```

## 步骤 9：广播给所有 WebSocket 客户端

### 执行主体

`server/routes/chat.js` 的 `broadcast()` 函数

### 接收数据

步骤 8 中 `emitStreamEvent()` 产生的每个事件对象。

### 处理逻辑

```javascript
function broadcast(msg) {
  for (const client of clients) {
    wsSend(client, msg);
    // → client.send(JSON.stringify(msg))
  }
}
```

`clients` 是一个 `Set<WebSocket>`，包含当前所有连接的 WebSocket 客户端。本场景中只有桌面端的一个客户端。

### 输出数据

通过 WebSocket 发送的完整事件序列（按时间顺序）：

```
→ { type: "status", isStreaming: true }
→ { type: "mood_start", ... }
```

## 步骤 11：对话结束后的记忆处理

### 执行主体

多个模块协同：`chat.js` → `memory-ticker.js` → `session-summary.js` → `compile.js` → `deep-memory.js`

### 11a. turn_end 事件处理（chat.js）

当 `event.type === "turn_end"` 时：

```javascript
// chat.js hub.subscribe 回调

// 1. flush 所有 parser（确保 buffer 中残留内容输出）
ss.thinkTagParser.flush(...);
ss.moodParser.flush(...);
ss.xingParser.flush(...);

// 2. 广播 turn_end
emitStreamEvent(sessionPath, ss, { type: "turn_end" });

// 3. 结束流式记录
finishSessionStream(ss);

// 4. 重置 parser 状态
ss.isThinking = false;
ss.thinkTagParser.reset();
ss.moodParser.reset();
ss.xingParser.reset();

// 5. 尝试生成会话标题（首轮对话）
maybeGenerateFirstTurnTitle(sessionPath, ss);
```

**标题生成**（异步，不阻塞）：

### 11b. notifyTurn（memory-ticker.js）

在步骤 5 中，`session.prompt()` 完成后调用了 `memoryTicker.notifyTurn(sessionPath)`：

```javascript
notifyTurn(sessionPath) {
  this._turnCount++;  // turnCount = 1

  // 1 % 6 !== 0 → 不触发滚动摘要
  // 首轮对话不会立即产生记忆
  // 要到第 6 轮对话才会触发
}
```

**但是**，当用户关闭这个 session 或切换到新 session 时，会触发 `notifySessionEnd(sessionPath)`，此时会执行完整的记忆流水线。

## 步骤 12：下次对话时记忆生效

### 执行主体

`core/agent.js` 的 `buildSystemPrompt()` 方法

### 接收数据

从文件系统读取更新后的 `memory.md`：

```markdown

## 今天

用户今天的工作：上午完成了3个接口开发，下午修复了2个bug。整体节奏合理，上午专注开发、下午转入维护。

（暂无）

## 长期情况

（暂无）
```

### 处理逻辑

`buildSystemPrompt()` 将 `memory.md` 注入到 System Prompt 的"记忆"段。下次用户发消息时，AI 模型会看到这段记忆。

### 输出数据

System Prompt 中的记忆段变为：

```markdown
---

# 记忆

## 记忆使用规则

记忆和用户档案是你内化的背景知识...

以下这些是从过往对话积累的记忆。

## 重要事实

（暂无）

## 今天

用户今天的工作：上午完成了3个接口开发，下午修复了2个bug。整体节奏合理，上午专注开发、下午转入维护。

## 最近一周

（暂无）

## 长期情况

（暂无）
```

### 实际效果

如果用户接下来说 `"今天效率怎么样？"`，Agent 不会说"根据我的记忆"，而是自然地回应：

> 今天节奏挺好的——上午 3 个接口、下午 2 个 bug，开发和维护都有推进。如果接口复杂度不低的话，这个产出很扎实了。

记忆的影响是**无声的**：Agent 知道用户今天做了什么，但不会暴露"记忆"这个机制的存在。
