# 关键设计决策

本文档记录 OpenHanako 中一些重要的架构和设计选择，以及它们背后的原因。

---

## 1. 为什么用 SQLite FTS5 而不是向量数据库？

### 选择

记忆搜索使用 SQLite 的 FTS5 全文搜索 + 标签匹配，而非 Pinecone、Weaviate 等向量数据库。

### 原因

| 维度 | SQLite FTS5 | 向量数据库 |
|------|-------------|------------|
| 部署 | 零依赖，随应用分发 | 需要独立服务或云服务 |
| 成本 | 免费 | 通常收费 |
| 延迟 | 本地查询，毫秒级 | 网络请求，百毫秒级 |
| 复杂度 | 简单，一个 .db 文件 | 需要管理 embedding 模型、索引 |
| 准确性 | 关键词匹配，精确但不语义 | 语义搜索，更智能但可能误匹配 |
| 离线 | 完全离线 | 通常需要网络 |

对于个人助理场景，记忆量通常在千条级别，FTS5 完全够用。标签系统弥补了语义搜索的不足。

## 2. 为什么 Server 是独立进程而不是嵌入主进程？

### 选择

Server 通过 `child_process.fork()` 运行在独立进程中。

### 原因

1. **崩溃隔离** — Server 崩溃不会导致 Electron 窗口消失，可以自动重启
2. **独立运行** — 不需要 Electron 也能运行（`npm run server`），支持 CLI 模式
3. **调试方便** — 前端通过标准 HTTP/WS 协议通信，可以用 Postman 等工具调试
4. **资源隔离** — AI 推理和工具执行不会阻塞 UI 渲染

---

## 3. 为什么记忆注入时要求"存在感为零"？

### 选择

System Prompt 中明确要求 Agent 不要说"我记得"、"你之前说过"等表述。

### 原因

这是一个 **UX 设计决策**，而非技术限制：

1. **自然感** — 朋友之间不会说"根据我的记忆数据库"，而是自然地运用已知信息
2. **避免尴尬** — 如果记忆过时或错误，说"我记得"会让用户困惑
3. **减少干扰** — 用户不需要知道 Agent 有"记忆系统"这个概念
4. **信任建立** — 通过行为而非言语展示记忆能力，更有说服力

---

## 4. 为什么用 Thin Facade 模式？

### 选择

HanaEngine 是一个 Thin Facade，不包含业务逻辑，只委托给内部 Manager。

### 原因

1. **避免 God Object** — 如果所有逻辑都在 Engine 中，文件会膨胀到数千行
2. **职责单一** — 每个 Manager 只负责一个领域
3. **可测试** — Manager 可以独立测试，不需要初始化整个 Engine
4. **依赖注入** — 通过构造器注入依赖，避免循环引用

### 依赖注入示例

```javascript
this._agentMgr = new AgentManager({
  getPrefs: () => this._prefs,        // 延迟获取，避免初始化顺序问题
  getModels: () => this._models,
  getHub: () => this._hub,            // Hub 在 Engine 之后创建
  getEngine: () => this,
});
```

## 5. 为什么沙盒有三层防御？

### 选择

沙盒系统有三层：Preflight → PathGuard → OS 沙盒。

### 原因

```
Preflight（体验层）
  │  拦截明显危险命令（sudo 等）
  │  给用户友好的错误提示
  │  Windows 上是主要安全层
  │
  ▼
PathGuard（应用层）
  │  路径级别的访问控制
  │  所有平台通用
  │  处理符号链接等边界情况
  │
  ▼
OS 沙盒（内核层）
  │  seatbelt / bwrap
  │  即使应用层被绕过，OS 层兜底
  │  网络隔离
```

**纵深防御**：任何单一层被绕过，其他层仍然提供保护。

## 6. 为什么前端是 React + 旧 Vanilla JS 混合？

### 选择

项目正在从 Vanilla JS 迁移到 React，通过 bridge.ts 让两套代码共存。

### 原因

1. **渐进式迁移** — 一次性重写风险太大，逐步迁移更安全
2. **Proxy 桥接** — 旧代码的 state 通过 Proxy 映射到 Zustand，读写透明
3. **Shim 层** — 为已迁移的模块提供兼容接口，旧代码无需修改
4. **优先级** — 核心功能优先，UI 迁移可以慢慢来

## 7. 为什么用 JSONL 而不是数据库存储对话？

### 选择

对话记录存储为 JSONL 文件（每行一个 JSON 对象），而非数据库。

### 原因

1. **简单** — 不需要额外的数据库连接和 schema
2. **可读** — 可以直接用文本编辑器查看
3. **流式写入** — 每条消息追加一行，不需要事务
4. **Pi SDK 兼容** — Pi SDK 的 SessionManager 使用 JSONL 格式
5. **备份方便** — 直接复制文件即可

---

## 8. 为什么 MOOD/PULSE/REFLECT 用 XML 标签而不是 JSON？

### 选择

AI 的内心活动用 `<mood>...</mood>` XML 标签包裹。

### 原因

1. **LLM 友好** — 大语言模型对 XML 标签的遵循度很高
2. **流式解析** — XML 标签有明确的开始和结束标记，便于流式解析
3. **不干扰内容** — 标签内容是自由文本，不需要转义
4. **前端分离** — 前端可以轻松将标签内容从正文中分离出来

---

## 9. 为什么心跳间隔是 17 分钟？

### 选择

心跳巡检默认每 17 分钟触发一次。

### 原因

1. **避免整点** — 17 是质数，避免与其他定时任务（如每 5 分钟、每 15 分钟）重叠
2. **平衡频率** — 太频繁浪费 API 调用，太稀疏错过重要变化
3. **可配置** — 用户可以通过 `config.yaml` 的 `desk.intervalMinutes` 调整

---

## 10. 为什么技能用 Markdown 而不是代码？

### 选择

技能定义为 SKILL.md 文件，用 Markdown 描述。

### 原因

1. **LLM 原生** — Markdown 是 LLM 最擅长理解和生成的格式
2. **人类可读** — 任何人都能阅读和编辑技能
3. **安全** — Markdown 不是可执行代码，不会引入安全漏洞
4. **灵活** — 可以描述任何类型的行为模式，不受编程语言限制
5. **可组合** — 多个技能可以同时启用，自然地融合在 System Prompt 中

---

## 11. 为什么记忆编译有指纹缓存？

### 选择

每次编译前计算输入的 MD5 指纹，指纹未变则跳过。

### 原因

1. **节省 API 调用** — 编译需要调用 LLM，有成本
2. **避免重复** — 没有新对话时，记忆内容不变，无需重新编译
3. **幂等性** — 相同输入产生相同输出，缓存是安全的
4. **启动加速** — 启动时如果记忆未变，跳过编译可以加快启动

---

## 12. 为什么 Agent 间通信有防循环机制？

### 选择

AgentMessenger 有 maxRounds、`<done/>` 标记和冷却期。

### 原因

两个 Agent 互相发消息可能形成无限循环：

```
Agent A: "你好"
Agent B: "你好，有什么事？"
Agent A: "没什么，就是打个招呼"
Agent B: "好的，那我也跟你打个招呼"
Agent A: "谢谢，你真好"
... (无限循环)
```

防循环机制：
1. **maxRounds** — 最多 N 轮对话后强制结束
2. **`<done/>`** — Agent 可以主动说"对话结束"
3. **冷却期** — 同一对 Agent 之间有最小间隔
