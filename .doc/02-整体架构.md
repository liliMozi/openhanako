## 数据流：一次对话的完整旅程

以用户在桌面端发送一条消息为例，数据经过以下完整流程：

```
1. 用户在 InputArea 输入消息，点击发送
   │
   ▼
2. InputArea 通过 WebSocket 发送 { type: "prompt", text: "..." }
   │
   ▼
3. server/routes/chat.js 的 WebSocket handler 接收消息
   │
   ▼
4. hub.send({ text, role: "owner" }) 路由消息
   │  ├── 判断 role === "owner" 且无 sessionKey
   │  └── 调用 engine.prompt(text)
   │
   ▼
5. core/session-coordinator.js 的 prompt() 方法
   │  ├── 检查是否有活跃 session
   │  └── 调用 session.prompt(text)
   │
   ▼
6. Pi SDK createAgentSession 处理
   │  ├── 读取 agent.systemPrompt（包含身份+记忆+技能）
   │  ├── 调用 AI 模型（通过 lib/llm 或 SDK 内部）
   │  └── 流式输出 events
   │
   ▼
7. session.subscribe() 捕获流式事件
   │  ├── text_delta → 文字片段
   │  ├── tool_start/tool_end → 工具调用
   │  └── message_update → 消息更新
   │
   ▼
8. core/events.js 解析特殊标签
   │  ├── ThinkTagParser: <think>...</think> → think_start/think_text/think_end
   │  ├── MoodParser: <mood>...</mood> → mood_start/mood_text/mood_end
   │  └── XingParser: <xing title="...">...</xing> → xing_start/xing_text/xing_end
   │
   ▼
9. server/routes/chat.js 将事件广播给所有 WebSocket 客户端
   │  ├── { type: "text_delta", delta: "..." }
   │  ├── { type: "mood_start" } / { type: "mood_text" } / { type: "mood_end" }
   │  ├── { type: "thinking_start" } / { type: "thinking_text" } / { type: "thinking_end" }
   │  └── { type: "tool_start", name: "..." } / { type: "tool_end" }
   │
   ▼
10. desktop 前端接收 WebSocket 消息，实时渲染
    │  ├── 文字 → 追加到聊天气泡
    │  ├── mood → 折叠的内心活动区域
    │  ├── thinking → 思考指示器
    │  └── tool → 工具调用卡片
    │
    ▼
11. 对话结束后（turn_end）
    │  ├── memory-ticker.notifyTurn() → 每 6 轮触发滚动摘要
    │  ├── 摘要 → compile → memory.md → 刷新 system prompt
    │  └── deep-memory → 提取元事实 → 写入 facts.db
    │
    ▼
12. 下次对话时，agent.buildSystemPrompt() 会包含更新后的记忆
```

## 启动流程

### Electron 模式

```
npm start
  │
  ▼
desktop/main.cjs
  │
  ├── 1. 创建 Splash 窗口（启动画面）
  │
  ├── 2. first-run 检查
  │      └── 首次运行：创建 ~/.hanako/ 目录结构，创建默认 Agent
  │
  ├── 3. fork("server/boot.cjs")
  │      │
  │      ▼
  │    server/boot.cjs
  │      │
  │      ▼
  │    server/index.js
  │      ├── 创建 Fastify 实例
  │      ├── 创建 HanaEngine
  │      ├── engine.init() — 初始化所有 Agent、模型、技能
  │      ├── 创建 Hub — 消息调度
  │      ├── 创建 BridgeManager — 外部平台
  │      ├── 注册所有路由
  │      ├── 启动 HTTP 监听
  │      └── 发送 IPC { type: "ready", port, token }
  │
  ├── 4. 收到 ready 消息
  │      ├── 关闭 Splash
  │      └── 检查 isSetupComplete
  │            ├── 否 → 打开 Onboarding 窗口
  │            └── 是 → 创建主窗口
  │
  └── 5. 主窗口加载前端
         ├── Vite dev 模式：http://localhost:5173
         └── 生产模式：dist-renderer/index.html
```

### 独立服务端模式

```
npm run server
  │
  ▼
scripts/launch.js (mode: "server")
  │
  ▼
server/index.js
  ├── 同上初始化流程
  ├── 启动 HTTP 监听
  └── 启动 CLI (server/cli.js)
       └── WebSocket 连接到自身，提供终端交互
```

## 进程模型

```
┌──────────────────────────────────────────────────┐
│  Electron 主进程 (main.cjs)                       │
│  - 窗口管理                                       │
│  - IPC 处理                                       │
│  - 嵌入式浏览器 (WebContentsView)                  │
│  - 文件系统操作                                    │
└──────────────┬───────────────────────────────────┘
               │ fork
┌──────────────▼───────────────────────────────────┐
│  Server 子进程 (server/boot.cjs)                   │
│  - Fastify HTTP + WebSocket                       │
│  - HanaEngine (core/)                             │
│  - Hub (hub/)                                     │
│  - BridgeManager (lib/bridge/)                    │
│  - 所有业务逻辑                                    │
└──────────────────────────────────────────────────┘
               │ HTTP/WS
┌──────────────▼───────────────────────────────────┐
│  Renderer 进程 (React 前端)                        │
│  - 聊天 UI                                        │
│  - 设置页面                                       │
│  - 编辑器窗口                                      │
└──────────────────────────────────────────────────┘
```

关键设计：**主进程不直接处理业务逻辑**，所有 AI 相关操作都在 Server 子进程中完成。主进程只负责窗口管理和系统集成。这样做的好处是：

1. Server 崩溃不会导致整个应用退出（有崩溃重启机制）
2. 可以独立运行 Server（不需要 Electron）
3. 前端通过标准 HTTP/WS 协议与后端通信，方便调试
