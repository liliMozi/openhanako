## 一、目标

在 OpenHanako 中实现群聊功能：用户创建群聊 → 邀请多个 Agent（半夏/明鉴/素月）→ 用户发送消息 → **三人按顺序依次回复**（A 说完 B 看到 A 的内容再回复，C 看到 A+B 再回复）→ 支持 `@半夏` 点名强制优先回复。

每个 Agent 在群聊中拥有与 1:1 聊天完全相同的完整能力：人格（personality.md）、技能（skills）、工具（读写文件/搜索/浏览器/命令）、记忆（自动写入 `_memoryTicker.notifyTurn()`）、模型配置（各自独立）。

## 二、架构

### 9 字总结

**群聊 Tab（iframe）→ WebSocket → Engine → Agent；前端编排循环**

### 详细数据流

```
点击「群聊」Tab
  → PluginPageView 渲染 iframe（plugins/agent-groupchat/page/groupchat.html）
  → iframe 建立独立 WebSocket 连接到 server（ws://127.0.0.1:{PORT}/ws?token=...）
  → Token 由 routes/page.js 从 {hanakoHome}/server-info.json 读取注入 window.__TOKEN__
  → PORT 同样注入为 window.__PORT__
  → agents 目录注入为 window.__AGENTS_DIR__

用户发送 "大家好"
  → iframe 调用 send()
  → 手动渲染用户消息（一次）
  → http POST /groupchat/append 写入 GroupStore
  → for agentId of members:
      sessionPath = "{agentsDir}\{agentId}\sessions\group_{groupId}.jsonl"
      ws.send({ type:"prompt", text, sessionPath })
      waitTurn(sessionPath) → addEventListener("message") 监听 text_delta + turn_end
      http POST /groupchat/append 写入 Agent 回复
  → 下个 pick() 时 loadHistory() 读取 GroupStore

WS 发送后的服务端链路：
  ws.send({type:"prompt", sessionPath:"...agents/hanako/sessions/group_xxx.jsonl"})
  → server/routes/chat.js → hub.send(text, {sessionPath})
  → 路由表: 桌面 owner（无 sessionKey）→ submitDesktopSessionMessage
  → engine.ensureSessionLoaded(sessionPath)  ← 加载到缓存
  → engine.promptSession(sessionPath, text)  ← 触发 LLM
  → Pi SDK session 创建 → extensions/context.js 监听 session_start → 注入群聊历史
  → Agent 回复 → EventBus 广播 text_delta × N, turn_end
  → 所有 WS 客户端收到广播 → iframe 的 onmessage 收集 text_delta
```

## 三、插件文件清单与作用

```
plugins/agent-groupchat/
├── manifest.json              # 声明 page 贡献 + full-access 权限
├── index.js                   # 插件入口（onload 生命周期）
├── lib/
│   └── group-store.js         # 群聊数据持久化（JSONL 文件）
├── extensions/
│   └── context.js             # Pi SDK extension — 注入群聊上下文到 system prompt
├── routes/
│   ├── page.js                # GET /api/plugins/agent-groupchat/groupchat → HTML 页面
│   └── api.js                 # REST API（list/create/messages/append）
└── page/
    └── groupchat.html         # 前端 SPA（群聊 UI + WebSocket 客户端 + 多 Agent 编排）
```

### 各文件详细说明

#### manifest.json（8 行）

```json
{ "id": "agent-groupchat", "trust": "full-access", "contributes": { "page": { "title": "群聊", "route": "/groupchat" } } }
```

- `id` 决定插件在 PluginManager 中的标识
- `trust: "full-access"` 赋予完整权限（routes、extensions、lifecycle 等）
- `contributes.page` 在顶部 Tab 栏注册「群聊」选项卡

#### index.js — 插件入口（87 行）

**`onload()` 生命周期**：
1. 从 `this.ctx` 获取 `bus`（真实 EventBus 实例）、`dataDir`（`plugin-data/agent-groupchat/`）、`log`（带前缀的 logger）
2. 创建 `GroupChatStore` 实例 → 持久化群聊数据到 `plugin-data/agent-groupchat/groups/`
3. 调用 `setGroupStore(store)` 通过 `global.__groupchat_store` 共享给 extensions（避免 freshImport 双实例问题）
4. 注册 5 个 Bus handler：

| handler 名 | 触发方 | 功能 |
|---|---|---|
| `groupchat:send` | routes/api.js | 编排多 Agent 顺序回复（for...of + waitForAgentDone） |
| `groupchat:create` | routes/api.js | 创建群聊（写入 meta JSON） |
| `groupchat:append` | routes/api.js | 追加单条消息（轻量写操作） |
| `groupchat:list` | routes/api.js | 列出所有群聊 |
| `groupchat:messages` | routes/api.js | 读取群聊历史（支持 since 增量） |

**注意**：`groupchat:send` 内的 `bus.request("session:send")` 已弃用（因 R2.5 发现 session not in cache 问题）。实际群聊消息通过 WebSocket 发送。

#### lib/group-store.js — 持久化（64 行）

存储路径：`plugin-data/agent-groupchat/groups/`

- `createGroup({ id, name, members })` → 写入 `{id}.json`（元数据）
- `getGroup(groupId)` → 读取 `{id}.json`
- `listGroups()` → 扫描所有 `.json` 文件
- `append(groupId, message)` → 追加一行到 `{id}.jsonl`（JSONL 格式）
- `getMessages(groupId, since)` → 读取 JSONL，支持 `since` 时间戳增量

#### extensions/context.js — 上下文注入（36 行）

Pi SDK extension 工厂函数。PluginManager 在 Agent session 创建时调用。

`pi.on("session_start")`：检测 sessionPath 是否包含 `group_` → 从 `global.__groupchat_store` 读取群聊历史 → 拼接到 `ctx.agent.state.systemPrompt` 前面。

**关键**：使用 `global` 而非模块级变量，因为 `fresh-import.js` 每次 import 带不同 timestamp，会导致多实例（R4 修复）。

#### routes/page.js — HTML 注入（37 行）

`GET /api/plugins/agent-groupchat/groupchat`：
1. 读取 `page/groupchat.html`
2. 从 `{hanakoHome}/server-info.json` 读取 `token` 和 `port`
3. 计算 `agentsDir = {hanakoHome}/agents`
4. 注入到 HTML：`window.__TOKEN__`、`window.__PORT__`、`window.__AGENTS_DIR__`
5. 设置 `Cache-Control: no-store` 防止缓存

#### routes/api.js — REST API（40 行）

| 端点 | 方法 | 作用 |
|------|------|------|
| `/groupchat/list` | GET | 列出群聊 → `bus.request("groupchat:list")` |
| `/groupchat/create` | POST | 创建群聊 → `bus.request("groupchat:create")` |
| `/groupchat/messages` | GET | 获取消息 → `bus.request("groupchat:messages")` |
| `/groupchat/append` | POST | 追加消息 → `bus.request("groupchat:append")` |

#### page/groupchat.html — 前端 SPA（151 行）

单文件 HTML，含 CSS + JS。功能：

1. **主题融合**：`<link>` 加载 `hana-css` URL 参数指向的主题 CSS 文件
2. **WebSocket 客户端**：独立连接 + 自动重连
3. **多 Agent 顺序编排**：`send()` → for...of + `await waitTurn()` → 逐个触发
4. **群聊列表**：左侧栏渲染（仿 SessionList 风格）
5. **参与者面板**：左下角显示成员
6. **消息渲染**：三色圆形头像 + 名字 + 时间戳
7. **@mention 弹出菜单**（v15 加入）：输入 `@` 自动弹菜单选择助手
8. **历史加载**：`pick()` 时 `loadHistory()` 从 GroupStore 读取
9. **blockEvt 去重**：send() 期间阻止 WS 的 `session_user_message` 回显

## 五、持续未解决的关键问题

### 1. UI 与宿主不统一（核心障碍）

经过 15 轮迭代（硬编码颜色 → CSS 变量 → 频道 CSS 复刻），始终无法在 iframe 内完美复刻宿主 UI。

**根因**：iframe 中通过 `<link>` 加载的 `hana-css` 是宿主主题 CSS 的静态快照，变量路径不完整。`rgba(var(--accent-rgb), .12)` 等嵌套 CSS 函数在 iframe 中不可靠。`innerHTML += style="var(--xxx)"` 在 iframe 沙盒中 CSS 变量解析路径与宿主不同。

**已生效**：背景色（`--bg`）、文字色（`--text`/`--text-muted`）、强调色（`--accent`）、高亮背景（`--accent-light`）。

**未生效**：头像、输入框样式、`rgba()` 嵌套函数、部分 overlay 变量。

### 2. 输入框能力受限

iframe 中只能使用原生 `<textarea>`，无法使用宿主的 TipTap 富文本编辑器。这意味着：
- 无斜杠命令可视化菜单（`/skill`、`/plan` 等）
- 无模型切换器
- 无法上传附件（但助手本身拥有完整工具能力）

**注意**：助手的**回复能力不受影响**——它们走完整 Engine pipeline，拥有所有工具。

### 3. `groupchat:send` Bus handler 已实现但未生效

`index.js` 中的 `groupchat:send` handler 实现了完整的编排逻辑（for...of 循环 + `session:send` + `waitForAgentDone`），但因 R2.5 发现 `session:send` 路径不经过 `ensureSessionLoaded` 导致 session not in cache——**该 handler 保留但当前未被调用**。实际编排在前端 `send()` 中通过 WS 循环实现。
