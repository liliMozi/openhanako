## 一、Subagent 是什么

Subagent 是 Hanako 内置工具，允许一个 Agent **启动另一个 Agent** 执行独立任务，完成后将结果返回。

**关键特性**：
- 独立的 Pi SDK session（隔离的上下文、工具、记忆）
- 不污染父 session 的焦点状态
- 支持模型指定、工作目录继承
- 事件自动通过 EventBus 转发

## 四、executeIsolated 调用签名

```javascript
// 文件: core/session-coordinator.js L1596

engine.executeIsolated(prompt, {
    agentId,          // 目标 Agent ID (e.g. "hanako", "mingjian")
    cwd,              // 工作目录
    emitEvents: true, // 事件通过 EventBus 广播
    persist: path,    // 持久化目录（session JSONL 存这里）
    signal,           // AbortController.signal (超时控制)
    model,            // 模型覆盖
    toolFilter,       // 自定义工具白名单
    builtinFilter,    // 内置工具白名单
    subagentContext,  // true = 跳过记忆三段 + 团队名单
    onSessionReady,   // (sessionPath) => void  收到实际 sessionPath 后回调
});
```

**返回值**：`Promise<void>`（异步 fire-and-forget，结果通过 deferred store + steer 消息返回）

## 五、executeIsolated 内部流程

```javascript
// session-coordinator.js L1596-L1795

1. resolveAgentIdentity(agentId) → 确定执行者身份
2. 继承 parent session 的 cwd
3. SessionManager.open(sessionPath, sessionDir) → 打开/创建 session
4. createSession(mgr, cwd, memoryEnabled, model, { restore, agent, ... })
   → 加载 personality.md、skills、config.yaml
   → Pi SDK 初始化（工具栈 + 模型 + 记忆）
5. session.agent.prompt([{ role: "user", content: [{ type: "text", text: prompt }] }])
   → LLM 流式回复
6. Pi SDK 事件通过 engine._emitEvent(event, sessionPath) 广播
7. EventBus → chat.js hub.subscribe → 处理 → WS broadcast → 前端渲染
```

## 六、agentIdFromSessionPath 实现

```javascript
// agent-manager.js L567-L571

agentIdFromSessionPath(sessionPath) {
    const rel = path.relative(this._d.agentsDir, sessionPath);
    if (rel.startsWith("..")) return null;
    return rel.split(path.sep)[0] || null;
}
```

**示例**：
- `C:\...\agents\hanako\sessions\chat_1.jsonl` → `"hanako"`
- `C:\...\agents\mingjian\sessions\chat_2.jsonl` → `"mingjian"`
- `just_a_string` → `null`（不含在 agentsDir 下）

## 七、Subagent 工具中的关键 deps 注入

```javascript
// core/agent.js L422-L441 (createSubagentTool 的 deps 组装)

{
    executeIsolated: (prompt, opts) => this._cb.executeIsolated(prompt, opts),
    resolveUtilityModel: () => this._cb?.getCurrentModelId?.() || null,
    getDeferredStore: () => this._cb?.getDeferredResults?.(),
    getTaskRegistry: () => this._cb?.getTaskRegistry?.(),
    setSubagentController: ...,
    removeSubagentController: ...,
    getSessionPath: () => this._cb?.getCurrentSessionPath?.(),
    getParentCwd: () => this._cb?.getCwd?.() || null,
    listAgents: this._listAgents || null,
    currentAgentId: this.id,
    agentDir: this.agentDir,
    emitEvent: (event, sp) => this._cb?.emitEvent?.(event, sp),
    persistSubagentSessionMeta: ...,
}
```

**关键发现**：
- `this._cb.executeIsolated` → `engine.executeIsolated` → `sessionCoordinator.executeIsolated`
- `this._cb.emitEvent` → `engine._emitEvent` → `EventBus.emit`
- 因此引擎级代码可直接调 `engine.executeIsolated()` 和 `engine._emitEvent()`

## 八、方案 A 中我们 "抄袭" 了什么

`hub/group-executor.js` v2 中的 `streamAgentToSession` 函数：

```javascript
// 当时这样写的（L80-L105）
await engine.executeIsolated(prompt, {
    agentId,
    cwd: engine.getHomeCwd(agentId),
    emitEvents: true,
    persist: path.join(engine.agentsDir, agentId, "sessions", "group_sessions"),
    signal: controller.signal,
    onSessionReady: (sp) => { isolatedPath = sp; },
});
```

**这是对的**。问题出在：

1. `engine.subscribe` 不存在 → 应该用 `engine._eventBus.subscribe`
2. 事件转发 sessionPath 不匹配 → executeIsolated 的事件 sessionPath 是临时文件路径（如 `.../group_sessions/xxx.jsonl`），前端 `currentSessionPath` 是 `"group_chat"` → 不匹配 → 前端收不到

## 九、如果要在插件中用 executeIsolated

**问题**：插件拿不到 `engine` 实例。插件只有 `bus`（EventBus）。

**可能的路径**：
1. 通过 EventBus 发送事件 → Hub handler → engine.executeIsolated
2. 或者直接在 Hub 层实现（方案 A 的做法）

## 十、与当前插件方案的对比

| 维度 | 当前插件方案 (WS 循环) | executeIsolated 方案 |
|------|----------------------|---------------------|
| Agent 能力 | ✅ 完整 (走 submitDesktopSessionMessage) | ✅ 完整 (Pi SDK) |
| session 隔离 | ✅ 每个 Agent 独立 sessionPath | ✅ 自动管理 |
| 前端接收 | ✅ WS广播 text_delta → 手动渲染 | ⚠️ sessionPath 不匹配 |
| 代码复杂度 | 低 (~200行) | 中 (~300行) |
| 工具调用 | ✅ 完整 | ✅ 完整（可通过 filter 控制） |
| 记忆 | ✅ auto | ✅ auto |
| 终止/超时 | ✅ 180s timeout | ✅ 内建 AbortController |
